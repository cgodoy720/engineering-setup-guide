Forms & Surveys System

system overview — architecture, pages, data flow

The Forms & Surveys system provides a comprehensive form builder for creating, managing, and analyzing custom forms and surveys. It includes an Admin Dashboard for form creation and management (requires page:form_builder permission) and a Public Form Interface for respondents to complete forms. The system supports multiple question types, draft saving, analytics, and submission management.

2 — Admin vs Public
Admin Dashboard

Staff-only dashboard for creating, managing, and analyzing forms. Requires page:form_builder permission. Full CRUD operations, submission management, and analytics.

  • /forms Dashboard (list all forms)
  • /forms/new Create new form
  • /forms/:formId/edit Edit form
  • /forms/:formId/submissions View submissions
  • /forms/:formId/analytics Analytics
Public Form Interface

Public-facing form interface accessible via unique slug. No authentication required (rate limited). Question-by-question progression with progress tracking. Supports draft saving and resumption.

  • /form/:slug Public form view
  • Question-by-question progression
  • Progress tracking
  • Draft saving
  • Session management
3 — Admin Pages
Route Page Key Functionality
/forms FormBuilderDashboard List all forms (card/table views), filter, create/edit/duplicate/archive forms
/forms/new FormEditor Create new form with question editor, settings, live preview
/forms/:formId/edit FormEditor Edit existing form with question editor, settings, live preview
/forms/:formId/submissions FormSubmissions View submissions, flag submissions, add notes, export CSV/JSON
/forms/:formId/analytics FormAnalytics Stats, response distributions, completion rates, visualizations
4 — Public Pages
Route Page Key Features
/form/:slug PublicFormContainer Public form view, question-by-question progression, progress tracking, draft saving, session management
Question Types: The system supports 9 question types: text, long_text, email, number, scale, multiple_choice, checkbox, date, and file. Questions are stored as JSONB in the forms.questions column.
5 — API Layer
/api/forms (Admin)
  • POST / Create form
  • GET / Get all forms
  • GET /:formId Get form by ID
  • PUT /:formId Update form
  • DELETE /:formId Delete (archive)
  • POST /:formId/duplicate Duplicate form
  • PUT /:formId/status Update status
  • GET /:formId/submissions Get submissions
  • GET /:formId/analytics Get analytics
  • GET /:formId/export/csv Export CSV
  • GET /:formId/export/json Export JSON
All require page:form_builder
/api/public/forms
  • GET /:slug Get form by slug
  • POST /:slug/submit Submit form

  • GET /:slug/draft Get draft

No auth required, rate limited
Controllers

6 — Core Database Tables
2 tables support the forms system. Forms store questions and settings as JSONB, and submissions store responses as JSONB. See schema.sql for complete DDL.
forms
  • form_id UUID PRIMARY KEY
  • title Form title
  • description Form description
  • slug UNIQUE slug for public access
  • status Enum: draft, active, closed, archived
  • created_by Foreign key to users
  • expires_at Optional expiration
  • submission_limit Optional limit
  • submission_count Current count
  • settings JSONB (theme, redirect, etc.)
  • questions JSONB array of questions
form_submissions
  • submission_id UUID PRIMARY KEY
  • form_id Foreign key to forms
  • responses JSONB (question_id → answer)
  • respondent_email Optional email
  • submitted_at Submission timestamp
  • completion_time_seconds Time to complete
  • session_id Session tracking
  • ip_address IP address
  • user_agent Browser info
  • status Enum: completed, draft
  • notes Admin notes
  • flagged BOOLEAN flag
Question Structure: Questions are stored as JSONB arrays in forms.questions. Each question has: id, type, label, required, options (for multiple choice/checkbox), validation, and order. Responses are stored as JSONB objects mapping question IDs to answers in form_submissions.responses.
