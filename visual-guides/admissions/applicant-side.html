Applicant Side

all applicant-facing pages, components, and navigation

The applicant portal is a standalone set of routes outside the main app shell. It uses a separate auth system (applicant login vs. builder login). All pages share a consistent top bar with logo, welcome message, and navigation. The dashboard acts as the central hub, with progress cards that unlock based on the applicant's current stage.

2 — Applicant Signup
ApplicantSignup /apply/signup
Simple account creation form. Collects first name, last name, email, password, and confirm password. Password requires minimum 8 characters. On success, sends a verification email and redirects to the dashboard.
Form Fields
firstName, lastName, email, password, confirmPassword
API
POST /api/applications/signup
Connects to: /apply /login (sign in link)
3 — Applicant Dashboard
ApplicantDashboard /apply
The central hub showing a grid of 4 action cards (responsive: 1 col mobile, 2 tablet, 4 desktop). Each card represents a stage and shows different states based on the applicant's progress. Cards dynamically swap — the Pledge card replaces Workshop after acceptance, and the Onboarding card replaces Pledge after pledge completion.
Info Session Card
States: not signed-up → signed-up → attended. Shows event date/time/location when registered. Links to /info-sessions.
Application Card
States: not started → in process → submitted → ineligible. Shows progress (X/5 sections). Deferral UI. Links to /application-form.
Workshop Card
States: locked → not signed-up → signed-up → attended. Locked until invited. Shows event details. "Enter Workshop" button. Links to /workshops.
Pledge / Onboarding Card
Pledge: locked → not completed → completed. Locked until accepted. Swaps to Onboarding card after pledge. Shows task progress.
APIs: GET /api/admissions/applicants/:id/stage GET /api/applications/applicant/:id/application
4 — Info Sessions
InfoSessions /info-sessions
Grid of available info session event cards (1 col mobile, 2 tablet, 3 desktop). Each card shows date, time, location, capacity, and online/in-person status. Past events are filtered out. Applicants can register/cancel. Online sessions show a "Join Meeting" button.
Event Card
Date, time, location, capacity count, "Reserved" badge when registered, online meeting link
Actions
Register, cancel registration. Capacity tracking prevents over-registration.
API: GET /api/info-sessions POST /api/info-sessions/:id/register
5 — Application Form
ApplicationForm /application-form
Multi-section form with dynamic question rendering. Sections are loaded from the database. Supports conditional questions (show when parent equals specific value). Auto-saves every response with a 1-second debounce. An eligibility check runs when leaving the eligibility section — if the applicant fails, they're marked ineligible and get a rejection email.
Auto-Save
1-second debounce saves each response to POST /api/applications/response/anonymous. No data loss on navigation.
Eligibility Check
Validates age, income, location, work auth, commute, schedule. Uses POST /api/applications/check-eligibility.
Conditional Logic
Questions can depend on parent answers via parent_question_id and show_when_parent_equals.
Input Types
text, textarea, radio, checkbox, select, date, email, tel, number, info. Address autocomplete for address fields.
Progress Tracking
Section tabs with completion indicators. Overall progress bar (% complete). Validation with error highlighting.
Submission
PUT /api/applications/:id/submit/anonymous. Redirects to dashboard with success message.
Connects to: /apply (back / on submit)
6 — Workshops
Workshops /workshops
Grid of workshop event cards. Only accessible if the applicant has been invited (checks stage API on mount — redirects if not invited). Features a laptop selection modal during registration (needs laptop vs. has laptop). The "Enter Workshop" button is the gateway to the learning environment — it converts the applicant to a builder account and grants access to the workshop content.
Access Control
Checks GET /api/admissions/applicants/:id/stage. Must be at workshop_invited stage or later.
Laptop Selection
Modal during registration: "I need a laptop" vs "I have a laptop". Stored in event_registration.needs_laptop.
Enter Workshop
POST /api/workshop/enter converts applicant to builder. Checks access window and early access flags.
Event Cards
Date, time, location, capacity. Register/cancel. "Reserved" badge when registered.
Connects to: /apply (back) /dashboard (enter workshop)
7 — Pledge
Pledge /pledge
A 6-step wizard with a progress bar. Only available after program acceptance. Collects commitments and personal information, ending with a canvas-based signature that supports both mouse and touch input.
Step 1: Introduction
Welcome message explaining the pledge purpose
Step 2: Learning
Learning commitment acknowledgments
Step 3: Community
Community commitments + Code of Conduct checkbox (opens modal)
Step 4: Building
Adapting & Building commitment acknowledgments
Step 5: Information
Builder info form: name, email, phone (formatted), date of birth
Step 6: Signature
Canvas signature + printed name + date. Touch-enabled. Stored as pledge_signature_data.
API: POST /api/admissions/pledge/complete
8 — Onboarding
Onboarding /onboarding
An 8-step task wizard with a sidebar showing task progress. Each task has a checkbox that syncs with the backend. The "Create Builder Account" button enables when all required tasks are complete. This is the final step — it creates the builder account and transitions the applicant to the main application.
1. Program Details
Program details and requirements overview
2. Attendance
Attendance policy and calendar review
3. Slack
Join Slack and AI-Native Builder channel
4. Pursuit Email
Set up Pursuit email account
5. Google Calendar
Set up and connect Google Calendar
6. Kisi
Download and sign into Kisi for building access
7. Building in Public
Set up public building profile/presence
8. Tech News
Engage with tech news sources and communities
API: GET/POST /api/onboarding/tasks POST /api/onboarding/create-account
